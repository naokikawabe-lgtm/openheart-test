# 提案活動の原則

## 基本姿勢

### 1. 相手の現状と文脈を深く理解することがすべての起点
提案の質は、顧客理解の深さに比例する。
表面的な課題ではなく、その背景にある構造的な要因を把握する。

- 顧客の業界構造、競争環境、内部事情を調べる
- 公開情報だけでなく、ヒアリングで得た生の情報を重視する
- 「相手が言っていること」と「本当に困っていること」を区別する

### 2. 売り込むな、お客様の相談・共創の相手になれ
提案は「自社のサービスを売る行為」ではなく「顧客の課題を一緒に解く行為」である。

- 自社のソリューションありきで考えない
- 顧客にとって最善の選択肢を提示する（自社でなくてもよい場面もある）
- 長期的な信頼関係の構築を最優先する

### 3. 具体的であること
抽象的な提案は刺さらない。数字、事例、スケジュールで具体化する。

- 定量的な効果を必ず示す（推定でもよいが根拠を明示）
- 類似事例・実績を引用する
- 「いつまでに」「誰が」「何を」を明確にする

## 提案プロセス

### Phase 1: 情報収集・顧客理解
- 顧客の公開情報（IR、プレスリリース、業界レポート）を収集
- 過去の接点・やり取りを確認（`sources/` を参照）
- ヒアリング内容を整理し `data/clients/` に格納

### Phase 2: 課題の構造化
- 顧客の課題をMECEに分解する
- 根本原因（Root Cause）を特定する
- 優先順位を付ける（インパクト × 実現可能性）

### Phase 3: ソリューション設計
- 課題に対する解決アプローチを複数案出す
- 各案のメリット・デメリット・リスクを整理
- 推奨案を選定し、その理由を明確にする

### Phase 4: 資料作成
- `story-writing.md` のフレームワークでストーリーを組む
- Markdownで構成を固めてから、PPTX生成に進む
- 必ず第三者レビューを入れる

### Phase 5: 提案・フォローアップ
- 提案後のフィードバックを `meetings/` に記録
- 次のアクションを明確にし `summary.md` に反映
- 提案内容の改善点をスキルファイルにフィードバック

## アンチパターン（やってはいけないこと）
- 顧客の話を聞く前にソリューションを決め打ちする
- 汎用的なテンプレートをそのまま使い回す
- 定量的な根拠なしに「効果がある」と主張する
- フォローアップを怠り、提案して終わりにする
- 競合の悪口で差別化しようとする
